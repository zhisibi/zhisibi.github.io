解决在zz.ac平台上URL重定向丢失子目录和图片加载失败的问题

2026年01月01日 约838字 预计需要3分钟

今天我们解决的是一个非常经典的 “子目录部署（Sub-directory Deployment）” 案例。当博客从根目录 / 迁移到 /blog/ 时，会产生一系列路径连锁反应。

以下是针对 URL 重定向丢失子目录 和 图片加载失败 的完整技术总结：

1. 核心矛盾：为什么会出问题？

  • URL 丢失 /blog 根源：Web 服务器（如 Python 或默认 Caddy）在处理文件夹请求时，如果末尾没带斜杠（如访问 /blog/about），会自动重定向到带斜杠的地址。但因为它不知道自己在子目录下，会错误地跳到根部的 /about/。
  • 图片碎裂根源：Hexo 渲染时，如果你在主题配置里写的是绝对路径 /images/xxx.jpg，浏览器会跳过 /blog 目录，直接去域名的最顶层寻找资源。

2. URL 重定向问题的终极方案

解决方案：在 Caddy 中接管重定向逻辑。

我们弃用了不稳定的 Python http.server，改用 Caddy 容器化部署，并在 Caddyfile 中配置了：

  • uri strip_prefix /blog：内部剥离路径，让文件查找能对应到物理目录。
  • try_files {path} {path}/ /index.html：这是最关键的一行。它让 Caddy 在内部尝试寻找文件或文件夹，而不是发送一个会让浏览器跳出子目录的外部 301 重定向。

3. 图片不能加载的兼容性方案

解决方案：Caddy 路径“兜底”映射。

由于你的代码需要兼容多台服务器（有的在根目录，有的在子目录），我们没有强行修改 Hexo 源码，而是让 Caddy 做了适配：

  • 静态文件夹句柄：添加了 handle /images/*、handle /css/* 等配置。
  • 效果：即便 HTML 里的图片链接是 /images/avatar.jpg（没有 /blog），Caddy 也会拦截这个请求并正确导向 public_html/images/ 目录。
  • 好处：这套 Hexo 代码部署到任何地方都不需要改动。

4. 部署与运维的最佳实践

为了保证博客长期稳定，我们完成了以下架构升级：

  • 容器化 (Podman)：
    • 使用 docker.io/library/caddy 镜像。
    • 通过 -v 挂载宿主机的静态文件和配置文件，实现了“代码与环境隔离”。
    • 使用 --network host 或 --add-host 解决了容器访问宿主机端口（如 8083 端口服务）的通信问题。
  • 自动化服务 (Systemd)：
    • 利用 podman generate systemd 生成了系统服务。
    • 通过 systemctl --user enable --now 实现了开机自启。
    • 开启 loginctl enable-linger 保证 SSH 退出后服务不断开。

5. 避坑清单（给未来的备忘录）

  1. 浏览器缓存是“魔鬼”：301 重定向会被浏览器死死记住。调试路径问题时，必须开启 无痕模式 或在开发者工具中勾选 Disable Cache。
  2. Hexo 链接规范：在主题菜单配置中，尽量使用 about/ (相对路径) 而不是 /about/ (绝对路径)，这样 Hexo 的 url_for 插件才能根据 root 配置自动补全。
  3. 容器内外路径：在 Caddyfile 里的 root 路径必须是容器内的挂载路径（如 /usr/share/caddy/...），而不是宿主机的路径。

最后建议： 既然你的 Caddy 已经跑通了反向代理，以后如果你有新的项目（比如 8083 端口的服务），只需要在 Caddyfile 里增加一个 handle_path /new-app* { reverse_proxy ... } 即可，非常方便扩展。
